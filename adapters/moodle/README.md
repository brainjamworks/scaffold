# Scaffold Moodle Adapter

Native Moodle activity module for Scaffold.

The adapter is intentionally thin:

- `scaffold/` is the Moodle plugin directory. Install or symlink it as
  `mod/scaffold` in a Moodle checkout.
- `frontend/src/` coordinates the Moodle outer page and mounts
  `@scaffold/core` inside one isolated inner document.
- `scaffold/public/` is generated by
  `vp run @scaffold/adapter-moodle#build`.

## Build

```sh
vp run @scaffold/adapter-moodle#verify
```

The verification command checks generated artifact freshness, lints the PHP
package, runs the standalone TypeScript and PHP suites, and builds the frontend.
The adapter's `test` and `build` commands also reject artifact drift before they
run.

Canonical assessment and learner-activity contracts live in
`packages/contracts`; canonical grading behavior and its conformance corpus live
in `packages/grading`. Moodle consumes byte-identical generated copies:

- installable schemas in `scaffold/schemas/` for the PHP validators;
- the standalone grading corpus in `tests/fixtures/` for PHP parity checks.

Do not edit those copies by hand. After changing and regenerating their package
sources, run the applicable sync command and commit the resulting tracked copy:

```sh
vp run @scaffold/adapter-moodle#sync:assessment-artifacts
vp run @scaffold/adapter-moodle#sync:learner-activity-artifact
vp run @scaffold/adapter-moodle#check:artifacts
```

Moodle-native PHPUnit sources are shipped under `scaffold/tests/`. Running
them requires a separately initialized, supported Moodle PHPUnit environment;
the adapter's standalone `verify` command checks and packages their source but
does not initialize Moodle or claim installed-system coverage.

## Installed Verification

Build the installable candidate before synchronizing it into Moodle. Keep the
checkout root, public webroot, PHPUnit wrapper, and plugin directory explicit;
Moodle releases may not place the public webroot at the checkout root.

```sh
vp run @scaffold/adapter-moodle#build
rsync -a --delete adapters/moodle/scaffold/ "$MOODLE_PLUGIN_DIR/"
php "$MOODLE_CHECKOUT/admin/cli/upgrade.php" --non-interactive
php "$MOODLE_CHECKOUT/admin/cli/purge_caches.php"
"$MOODLE_PHPUNIT" --configuration "$MOODLE_WEBROOT/phpunit.xml" \
  --testsuite mod_scaffold_testsuite
```

The candidate is not installed byte-for-byte until Moodle's ignored AMD build
and `scaffold/public/` outputs have also been generated and synchronized. Do
not run Moodle's PHPUnit initialization command as routine verification: it
recreates the configured test database and dataroot.

Installed verification covers both the declared minimum Moodle release and the
newest supported release. Use disposable databases and dataroots for clean
install, cron, privacy, and backup/restore checks. The complete
repository gate remains:

```sh
vp run verify:release
```

## Backend Lifecycle

- Authored content, learner-safe content, assessment targets and groups, grade
  configuration, and Scaffold media belong to the activity. Assessment state
  and learner-activity state are separate per-user records.
- Standalone assessment commands accept standalone-owned targets only. Quiz
  child targets are mutated through quiz attempt commands, and terminal quiz
  attempts are the grade authority for graded quiz groups.
- Assessment mutations commit canonical state before completion or gradebook
  projection. A gradebook failure therefore does not ask the learner to submit
  again; the response still reports the saved assessment outcome.
- Timed quiz expiry is reconciled on learner access and by the scheduled expiry
  task. Grade items and learner grades are rebuilt from current authored
  definitions and canonical learner state by the scheduled grade task.
- Retryable grade failures are retried in bounded cron work. Locked,
  configuration-error, and retry-exhausted rows appear on the capability-gated
  Grade publication status page with privacy-safe codes and an explicit,
  confirmed requeue action.
- The privacy provider discovers, exports, and deletes assessment state,
  learner-activity state, and learner publication status. Course-owned media is
  not removed by deletion of one user's personal data.

The two Scaffold scheduled tasks can be exercised independently in an
installed environment:

```sh
php "$MOODLE_CHECKOUT/admin/cli/scheduled_task.php" \
  --execute='\mod_scaffold\task\reconcile_quiz_expiry'
php "$MOODLE_CHECKOUT/admin/cli/scheduled_task.php" \
  --execute='\mod_scaffold\task\reconcile_assessment_grades'
```

## Backup and Restore

Moodle backup always includes the portable activity definition and media. A
backup without enrolled users restores no assessment, learner-activity,
publication, or completion rows. A backup with enrolled users restores the two
canonical learner-state domains and completion data; grade publication rows
and Moodle gradebook projections are rebuilt from restored authority.

Restore and duplication repair the course-module-derived artifact identity in
the activity and both canonical learner snapshots. Due quizzes are reconciled
against server time during restore, and completion and grades are rebuilt from
the repaired state. The same rules apply when the activity is imported into an
existing course.

## Adapter Seams

- Authoring and learner routes create exactly one Scaffold-owned iframe from
  the same generated `moodle-inner.html`; the inner document never creates a
  nested Scaffold iframe or loads Moodle theme CSS.
- The authenticated outer Moodle page owns login, capabilities, completion,
  route selection, the lightweight ESM loader, and `core/ajax`. After a
  validated `inner.ready` handshake it sends the already-authorized
  application configuration and proxies only the adapter's allowlisted
  `mod_scaffold_*` calls.
- The isolated inner document owns Core and KaTeX styles, the React lifecycle,
  and the bridge-backed Moodle ports. Messages are scoped by origin,
  source, protocol version, channel, session, kind, and payload shape.
- `view.php` always owns the learner surface and requests the learner-safe
  payload projection. Editing capability does not change that route's
  bootstrap contract.
- `author.php` is the dedicated authoring surface. Moodle checks
  `mod/scaffold:editcontent`, then the frontend explicitly requests the
  authoring payload projection. Capability and payload purpose are separate
  decisions.
- Core's `ScaffoldAuthoringEntry` owns first-document creation and the shared
  authoring chrome. Moodle supplies persistence and media services plus a
  small, accessible return link; it does not duplicate Core editor controls.
- The authoring route uses Moodle's embedded layout and a full-viewport frame;
  the contained editor is its scroll owner. Learner routes report the stable
  layout-root border-box height so Page content can grow and shrink without a
  nested scrollbar. Fullscreen measurements are ignored and the last inline
  height is restored on exit.
- Page and Slideshow use the same learner frame policy. Core selects the
  player, and Core alone owns Slideshow's single fullscreen entry and exit
  control; the shared iframe grants fullscreen permission.
- Persistence stores author JSON, learner-safe JSON, and projected assessment
  target contracts in Moodle tables.
- Collaboration is disabled by wiring a null provider. Moodle authoring is
  treated as a single-author surface for the first adapter.
- Media upload/resolve uses Moodle's File API in the `media` file area for
  managed upload-backed blocks and attachments.
- Assessment check/submit/reveal flows use Moodle external functions called
  through `core/ajax`.
- Official grades are written through Moodle's gradebook API and recovered by
  Moodle scheduled tasks when immediate publication cannot finish.

The plugin never loads frontend source files directly. Vite output under
`scaffold/public/` is generated by `vp run @scaffold/adapter-moodle#build`.
Moodle's official component build generates `scaffold/amd/build/` from
`scaffold/amd/src/`. Both directories are ignored deployment outputs: rebuild
them before synchronising `scaffold/` into a Moodle checkout, and do not
commit them. No Moodle visual-compatibility reset is needed because Moodle and
Scaffold styles live in separate documents.
